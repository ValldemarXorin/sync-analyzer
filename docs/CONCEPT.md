# Концепция платформы интеллектуального анализа и оптимизации синхронного взаимодействия между микросервисами

## 1. Проблематика

Микросервисная архитектура (МСА) стала стандартом разработки сложных распределённых систем. Однако переход от монолита к множеству взаимодействующих сервисов порождает новые классы проблем, связанных с сетевыми вызовами.

**Синхронные взаимодействия (REST, gRPC) являются основным источником:**

- **Высоких задержек** – каждый вызов добавляет сетевое время и время обработки на стороне вызываемого сервиса.
- **Каскадных отказов** – сбой в одном сервисе может заблокировать цепочку вызывающих.
- **Сложности диагностики** – распределённый характер делает практически невозможным отследить проблему без специальных инструментов.
- **Архитектурного «мусора»** – со временем появляются неоптимальные паттерны (N+1 запросы, циклические зависимости, избыточные вызовы), которые трудно обнаружить вручную.

Существующие инструменты (мониторинг, трассировка, логирование) помогают увидеть симптомы, но **не отвечают на вопросы «почему это происходит» и «что делать»**. Разработчик получает горы графиков, но не готовые рекомендации.

## 2. Цель проекта

Разработать инструмент, который встраивается в процесс тестирования микросервисных систем и предоставляет разработчикам **конкретные указания по оптимизации синхронных взаимодействий**, основанные на объективных метриках и выявленных архитектурных антипаттернах.

## 3. Задачи

- Создать лёгкую библиотеку-агент для Java-микросервисов, автоматически собирающую данные о REST и gRPC вызовах без изменения бизнес-логики.
- Обеспечить сквозную трассировку запросов (стандарт W3C TraceContext).
- Разработать серверную часть для приёма, хранения и анализа собранных данных.
- Предложить метрики, отражающие качество синхронного взаимодействия (latency, ошибки, нагрузка).
- Реализовать алгоритмы обнаружения архитектурных проблем (N+1, циклы, избыточные вызовы, глубина цепочек).
- Ввести **интегральный коэффициент**, сводящий разнородные метрики в единую оценку «здоровья» эндпоинта/сервиса.
- Генерировать **текстовые рекомендации** по устранению выявленных узких мест.
- Предоставить веб-интерфейс для визуализации трасс, дашбордов и списка проблем.

## 5. Уникальность предлагаемого подхода

1. **Синхронный фокус** – одновременная поддержка двух самых популярных протоколов синхронного взаимодействия (REST и gRPC) в рамках одного агента.
2. **Интегральный коэффициент** – новое качество: вместо набора разрозненных графиков даётся один понятный показатель, позволяющий быстро ранжировать проблемы.
3. **Рекомендательный движок** – не просто «что плохо», а «что делать» (например, «обнаружен N+1 запрос: используйте @BatchSize или JOIN FETCH»).
4. **Акцент на тестирование** – инструмент предназначен для запуска в тестовых средах и CI/CD, не требует сложной эксплуатации в production.
5. **Лёгкость интеграции для Java** – подключение через Spring Boot Starter одной зависимостью.
6. **Открытость для других языков** – простое HTTP API для отправки спанов.

## 6. Интегральный коэффициент: детали (будет уточнен в будущем)

Коэффициент рассчитывается для каждого эндпоинта (или агрегированно для сервиса) на основе шести компонент.

### 6.1. Компоненты и нормировка


| Компонента              | Обозначение | Нормировка (значение от 0 до 1)                             |
| ----------------------- | ----------- | ----------------------------------------------------------- |
| p95 latency             | L           | `min(1, p95_latency_ms / threshold_ms)` (threshold=500 мс)  |
| Error rate (5xx)        | E           | `min(1, error_rate / 0.1)`                                  |
| Throughput (RPS)        | T           | `min(1, current_rps / max_capacity_rps)` (max_capacity=100) |
| N+1 запросы             | N           | `1` если обнаружены за период, иначе `0`                    |
| Циклические зависимости | C           | `1` если сервис участвует в цикле, иначе `0`                |
| Глубина цепочки         | D           | `min(1, depth / max_allowed_depth)` (max_allowed_depth=5)   |


*Пороговые значения могут настраиваться пользователем.*

### 6.2. Веса

Веса назначаются экспертным методом (опрос 8–10 разработчиков/архитекторов). Текущий вариант:

- **L**: 0.35 (задержка важнее всего)
- **E**: 0.25 (ошибки критичны для пользователя)
- **T**: 0.10 (нагрузка сама по себе не проблема, но важна в контексте)
- **N**: 0.15 (N+1 сильно влияет на производительность)
- **C**: 0.10 (циклы могут приводить к бесконечным вызовам)
- **D**: 0.05 (глубина менее значима, если нет других проблем)

### 6.3. Итоговая формула

```
Score = 0.35*L + 0.25*E + 0.10*T + 0.15*N + 0.10*C + 0.05*D
```

### 6.4. Интерпретация

- **0.00 – 0.20**: отлично
- **0.21 – 0.40**: нормально
- **0.41 – 0.60**: требует внимания
- **0.61 – 0.80**: плохо
- **0.81 – 1.00**: критично

## 7. Целевые сценарии использования

### Сценарий 1: Разработчик перед релизом

- Подключает агент к своим микросервисам.
- Запускает интеграционные тесты.
- Открывает веб-дашборд, видит интегральные коэффициенты по эндпоинтам.
- Переходит в раздел «Проблемы», получает список с рекомендациями.
- Исправляет найденные узкие места до выхода в production.

### Сценарий 2: Архитектор оценивает систему

- Запускает инструмент на тестовом стенде со всей системой.
- Анализирует граф зависимостей, выявляет нежелательные циклы и избыточные связи.
- Получает рекомендации по рефакторингу.

### Сценарий 3: Тестировщик в CI/CD

- В пайплайне после прогона тестов автоматически запускается анализатор.
- Если интегральный коэффициент превышает порог (например, 0.6), пайплайн останавливается с отчётом.
- Разработчик получает уведомление о проблеме.

### **Сценарий 4: Эксплуатация и раннее выявление деградации**

- Платформа подключена к системе в режиме мониторинга.
- В процессе эксплуатации один из сервисов начинает постепенно увеличивать время ответа (рост p95 latency).
- Интегральный коэффициент для конкретного эндпоинта переходит из зоны 🟢 в 🟠.
- В разделе «Проблемы» фиксируется предупреждение о росте глубины цепочки вызовов и увеличении количества синхронных обращений к зависимому сервису.
- Команда получает уведомление (например, в Slack/Telegram).
- До возникновения критического сбоя проводится анализ и оптимизация — предотвращается каскадная деградация системы.

## 8. Границы проекта

- **Агент поддерживает только Java (Spring Boot)**. Для других языков предоставляется HTTP API.
- **Анализируются только синхронные вызовы** (REST, gRPC). Асинхронные (брокеры сообщений) не входят в первую версию.
- **Рекомендации строятся на правилах (эвристиках)**, а не на машинном обучении (но ML может быть добавлен в будущем).

## 9. Ожидаемые результаты

- Рабочий прототип, включающий агент, коллектор, анализатор, визуализатор.
- Подтверждение на демонстрационном стенде, что инструмент обнаруживает внесённые проблемы и даёт корректные рекомендации.
- Документация по использованию и развитию.
- Возможность использовать как базу для дальнейших исследований (например, автоматическое исправление, ML-модели).

## 10. Заключение

Предлагаемая платформа закрывает важный пробел между сырыми данными наблюдаемости и действиями разработчика. Интегральный коэффициент и рекомендательный движок превращают инструмент из пассивного наблюдателя в активного помощника, способного ускорить выявление и устранение проблем в микросервисной архитектуре.
