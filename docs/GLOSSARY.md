# Глоссарий терминов

## 1. Введение

Данный документ содержит определения ключевых терминов, используемых в проекте «Платформа интеллектуального анализа и оптимизации синхронного взаимодействия между микросервисами». Глоссарий предназначен для единообразного понимания терминологии всеми участниками проекта (разработчиками, тестировщиками, научным руководителем и членами комиссии).

---

## 2. Основные термины

### 2.1. Общие понятия

| Термин | Определение |
|--------|-------------|
| **Микросервисная архитектура (МСА)** | Архитектурный стиль, при котором приложение строится как набор небольших независимо развёртываемых сервисов, взаимодействующих по сети. |
| **Синхронное взаимодействие** | Способ взаимодействия, при котором вызывающий сервис блокирует выполнение и ожидает ответа от вызываемого сервиса. Основные протоколы: REST (HTTP) и gRPC. |
| **Асинхронное взаимодействие** | Способ взаимодействия, при котором вызывающий сервис не ждёт немедленного ответа (например, через очереди сообщений). В рамках проекта не рассматривается. |
| **Наблюдаемость (Observability)** | Свойство системы, позволяющее измерять её внутреннее состояние по внешним данным (метрики, логи, трассы). |

### 2.2. Компоненты платформы

| Термин | Определение |
|--------|-------------|
| **Агент** | Java-библиотека (Spring Boot Starter), подключаемая к микросервису для автоматического перехвата и сбора данных о синхронных вызовах. |
| **Коллектор** | Сервис, принимающий спаны от агентов и внешних сервисов, выполняющий их валидацию, обогащение и запись в Kafka. |
| **Анализатор** | Компонент, читающий спаны из Kafka, сохраняющий их в TimescaleDB, рассчитывающий метрики, выявляющий проблемы и генерирующий рекомендации. |
| **Визуализатор** | Веб-интерфейс для отображения собранных данных, метрик, проблем и рекомендаций. |
| **Платформа** | Совокупность всех компонентов (агент, коллектор, анализатор, визуализатор), обеспечивающая анализ и оптимизацию синхронных взаимодействий. |

## 3. Термины трассировки

### 3.1. Основные понятия

| Термин | Определение |
|--------|-------------|
| **Трассировка (Trace)** | Полная запись пути запроса через все микросервисы системы. Состоит из множества спанов. |
| **Спан (Span)** | Единица трассировки, представляющая одну логическую операцию (например, вызов REST-эндпоинта). Содержит временные метки начала и конца, атрибуты и идентификаторы. |
| **TraceId** | Глобально уникальный идентификатор всей трассировки (32 символа в hex). |
| **SpanId** | Уникальный идентификатор конкретного спана (16 символов в hex). |
| **ParentSpanId** | Идентификатор родительского спана. Для корневого спана отсутствует. |
| **Корневой спан** | Первый спан в трассировке, не имеющий родителя. Обычно соответствует входящему запросу на границе системы. |
| **W3C TraceContext** | Стандарт для передачи контекста трассировки между сервисами через HTTP-заголовки `traceparent` и `tracestate`. |

### 3.2. Виды спанов

| Термин | Определение |
|--------|-------------|
| **SERVER-спан** | Спан, соответствующий обработке входящего запроса на стороне сервера. |
| **CLIENT-спан** | Спан, соответствующий исходящему вызову клиента к другому сервису. |
| **REST-спан** | Спан для вызова по протоколу HTTP/REST. |
| **gRPC-спан** | Спан для вызова по протоколу gRPC. |

### 3.3. Атрибуты спана

| Термин | Определение |
|--------|-------------|
| **Service name** | Имя микросервиса, сгенерировавшего спан. |
| **Kind** | Тип спана: `SERVER` или `CLIENT`. |
| **Protocol** | Протокол взаимодействия: `REST` или `gRPC`. |
| **Method** | HTTP-метод (GET, POST и т.д.) или имя gRPC-метода. |
| **Path** | URI для REST или полное имя метода для gRPC. |
| **Start time** | Время начала операции (в формате ISO 8601). |
| **End time** | Время окончания операции. |
| **Duration** | Длительность операции в миллисекундах. |
| **Status code** | HTTP-статус (200, 404, 500) или gRPC-код. |
| **Error** | Сообщение об ошибке (при наличии). |
| **Tags** | Дополнительные метки (ключ-значение), например, `userId`, `dbQueryTime`. |

## 4. Метрики и анализ

### 4.1. Базовые метрики

| Термин | Определение |
|--------|-------------|
| **Latency (задержка)** | Время от начала до завершения операции. Измеряется в миллисекундах. |
| **p95 (95-й перцентиль)** | Значение, ниже которого находится 95% всех измерений. Используется для оценки «типичной» задержки без учёта выбросов. |
| **p99 (99-й перцентиль)** | Значение, ниже которого находится 99% всех измерений. Показывает «худшие» случаи. |
| **Error rate (доля ошибок)** | Отношение количества запросов, завершившихся ошибкой (коды 5xx), к общему числу запросов. |
| **Throughput (пропускная способность)** | Количество запросов в единицу времени (обычно RPS — requests per second). |
| **RPS** | Requests Per Second — количество запросов в секунду. |

### 4.2. Интегральный коэффициент

| Термин | Определение |
|--------|-------------|
| **Интегральный коэффициент** | Числовая метрика от 0 до 1, агрегирующая различные аспекты качества взаимодействия (latency, error rate, структурные проблемы). Позволяет быстро оценить «здоровье» эндпоинта или сервиса. |
| **Компоненты коэффициента** | Составляющие части интегрального коэффициента: L (latency), E (error rate), T (throughput), N (N+1), C (cycles), D (depth). Каждая нормируется к значению от 0 до 1. |
| **Вес компонента** | Коэффициент важности компоненты в итоговой формуле. Определяется экспертным методом. |

### 4.3. Архитектурные антипаттерны

| Термин | Определение |
|--------|-------------|
| **N+1 запрос** | Ситуация, когда сервис делает один запрос к родительской сущности, а затем N запросов к дочерним сущностям (по одному на каждую). Приводит к резкому росту числа вызовов и задержек. |
| **Циклическая зависимость (Cycle)** | Ситуация, когда сервисы вызывают друг друга по кругу (A → B → C → A). Увеличивает связанность и риск каскадных сбоев. |
| **Избыточные вызовы (Redundant calls)** | Повторяющиеся вызовы одного и того же эндпоинта с одинаковыми параметрами за короткое время, когда результат мог бы быть закеширован. |
| **Глубокая цепочка (Deep chain)** | Цепочка синхронных вызовов, превышающая рекомендуемую глубину (например, более 5 сервисов подряд). Увеличивает общую задержку и снижает надёжность. |
| **Антипаттерн** | Часто встречающееся ошибочное или неоптимальное решение в проектировании, приводящее к негативным последствиям. |

## 5. Технологии и инструменты

| Термин | Определение |
|--------|-------------|
| **Spring Boot** | Фреймворк для разработки Java-приложений, предоставляющий средства для создания микросервисов. |

| **TimescaleDB** | Расширение PostgreSQL, оптимизированное для работы с временными рядами. |
| **PostgreSQL** | Объектно-реляционная система управления базами данных. |
| **Docker** | Платформа для контейнеризации приложений. |
| **Docker Compose** | Инструмент для определения и запуска многоконтейнерных Docker-приложений. |
| **Thymeleaf** | Современный серверный шаблонизатор для Java. |
| **Bootstrap** | CSS-фреймворк для разработки адаптивных веб-интерфейсов. |
| **Chart.js** | JavaScript-библиотека для построения графиков. |

## 6. Инфраструктурные термины

| Термин | Определение |
|--------|-------------|
| **CI/CD** | Continuous Integration / Continuous Delivery — практики автоматизации сборки, тестирования и развёртывания. |
| **Docker Compose** | Инструмент для определения и запуска многоконтейнерных Docker-приложений. |

| **Топик (Topic)** | Категория или поток сообщений в Kafka. |

| **Гипертаблица (Hypertable)** | Основная абстракция TimescaleDB, автоматически партиционирующая данные по времени. |
| **Ретеншн (Retention)** | Политика хранения данных (срок, после которого данные удаляются). |
